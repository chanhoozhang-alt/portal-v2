# 南京分行应用管理系统 - 需求文档

## 一、系统定位

统一应用门户 + 应用角色人员管理平台。管理南京分行自己的应用，控制谁能看到什么应用、谁能在应用里担任什么角色。

## 二、用户进入方式

用户不在本系统输入账号密码。本系统作为“南京分行应用管理系统”接入上游统一认证平台，采用授权码模式完成登录。

当前流程为：

```text
用户从上游点击“南京分行应用管理系统”，默认访问本系统门户首页 /
  -> 前端默认调用 GET /api/portal/init
  -> 本系统发现未登录
  -> 302 跳转上游认证服务
  -> 上游认证成功后携带 code 回跳本系统
  -> 本系统后端用 code 换取 accessToken / refreshToken
  -> 本系统建立 PORTAL_SESSION 登录态
  -> 回到门户首页 /
  -> 前端再次调用 GET /api/portal/init 渲染页面
  -> 后续请求走本系统内部认证和权限判断
```

本系统不维护用户主数据（姓名、机构、部门等），全部从外部统一身份平台获取。

## 三、系统角色

| 角色 | 定位 | 能做什么 |
| --- | --- | --- |
| 系统管理员 | 平台最高权限 | 管理一切：所有应用、所有配置、所有人员 |
| 应用管理员 | 管理自己负责的应用 | 改应用配置、配应用管理员、配业务管理员、建自定义角色 |
| 业务管理员 | 只管角色人员 | 给自定义角色加人、减人 |
| 普通用户 | 最终使用者 | 只看首页、点应用跳转 |

多身份可叠加，取最高权限。

### 自定义角色的作用

本系统的自定义角色**不控制资源权限**（不控制菜单、按钮、接口）。用途有三个：

1. **控制应用可见范围** — `ROLE` 类型的应用，只有拥有该应用下某个角色的用户才能在首页看到
2. **维护人员分组** — 记录谁属于哪个应用的哪个角色
3. **给目标业务系统提供用户角色身份参考**

## 五、首页需求

### 1. 页面内容

- 按分组展示当前用户有权限访问的应用
- 支持应用搜索
- 点击应用卡片跳转到目标系统
- 记录用户操作行为（点击了哪个应用、什么时候点的）
- 管理员用户展示"后台管理"入口，普通用户不展示
- 当前登录人信息

### 2. 应用可见范围

| 枚举值 | 说明 | 适用场景 |
| --- | --- | --- |
| `ALL` | 全员可见 | 公共应用、通知公告类系统 |
| `ROLE` | 仅有角色员工可见 | 业务管理系统、敏感应用 |

### 3. 首页展示规则

首页只展示**状态正常**，并且当前用户满足以下**任一条件**的应用：

1. 应用可见范围 = `ALL`（全员可见）
2. 应用可见范围 = `ROLE`，且当前用户拥有该应用下有效自定义角色
3. 当前用户是该应用的应用管理员
4. 当前用户是该应用的业务管理员

> 系统管理员首页不默认展示全部应用，只展示实际可访问应用。进入后台后可管理全部应用。

### 4. 后台管理按钮展示规则

仅以下用户可见：系统管理员 / 应用管理员 / 业务管理员。普通用户不展示。

## 十六、首页可见应用 SQL 设计

### 核心判断逻辑

应用展示条件：**状态正常** 且满足以下任一条件：

1. 可见范围 = `ALL`
2. 可见范围 = `ROLE`，且用户拥有该应用下有效自定义角色
3. 用户是该应用的应用管理员
4. 用户是该应用的业务管理员

### 推荐 SQL

```sql
SELECT
    a.id,
    a.app_code,
    a.app_name,
    a.visible_type,
    a.status,
    a.sort_no,
    a.create_time
FROM app_info a
WHERE a.status = 'ENABLED'
  AND (
        a.visible_type = 'ALL'

        OR (
            a.visible_type = 'ROLE'
            AND EXISTS (
                SELECT 1
                FROM user_role ur
                WHERE ur.app_code = a.app_code
                  AND ur.user_id = :userId
                  AND ur.status = 'ENABLED'
            )
        )

        OR EXISTS (
            SELECT 1
            FROM app_admin aa
            WHERE aa.app_code = a.app_code
              AND aa.user_id = :userId
              AND aa.status = 'ENABLED'
        )

        OR EXISTS (
            SELECT 1
            FROM biz_admin ba
            WHERE ba.app_code = a.app_code
              AND ba.user_id = :userId
              AND ba.status = 'ENABLED'
        )
  )
ORDER BY a.sort_no ASC, a.create_time DESC;
```

### 推荐索引

```sql
CREATE INDEX idx_app_info_status_visible ON app_info(status, visible_type);
CREATE INDEX idx_user_role_user_app_status ON user_role(user_id, app_code, status);
CREATE INDEX idx_app_admin_user_app_status ON app_admin(user_id, app_code, status);
CREATE INDEX idx_biz_admin_user_app_status ON biz_admin(user_id, app_code, status);
```

## 十七、核心业务流程

### 1. 普通用户访问首页

> 用户登录 → 获取登录人信息 → 查询有效自定义角色 → 查询可见应用 → 展示应用列表 → 点击应用 → 跳转目标系统

### 2. 系统管理员进入后台

> 系统管理员登录 → 首页显示后台管理按钮 → 点击后台管理 → 可管理全部应用、分组、管理员、角色、角色人员

### 3. 应用管理员配置应用

> 应用管理员登录 → 首页显示后台管理按钮 → 进入应用管理 → 只能看到自己负责的应用 → 维护基础配置 / 自定义角色 / 业务管理员 / 角色人员

### 4. 业务管理员配置角色人员

> 业务管理员登录 → 首页显示后台管理按钮 → 进入角色人员配置 → 选择可管理的应用和角色 → 添加/移除角色人员 → 权限生效

## 十九、接口设计建议

### 1. 首页应用查询

```
GET /api/portal/apps
```

查询当前登录人可见应用。

### 2. 首页初始化（含后台权限判断）

> 原 `GET /api/admin/permission` 已合并到 `GET /api/portal/init`，一次性返回用户信息 + 权限 + 可见应用。

```
GET /api/portal/init
```

### 3. 应用列表

```
GET /api/admin/apps
```

## 二十、权限校验规则

### 1. 系统管理员

系统管理员 = 平台最高权限，可访问所有后台功能。

### 2. 应用管理员

操作时必须校验：**当前用户是否为该应用的应用管理员**。通过后可操作应用基础配置、应用管理员配置、业务管理员配置、自定义角色配置。

### 3. 业务管理员

操作时必须校验：**当前用户是否为该应用的业务管理员，或是否拥有该角色的人员管理权限**。通过后可操作查询/添加/移除角色人员。

不能操作：应用基础配置、应用管理员配置、业务管理员配置、自定义角色配置、系统管理员配置、应用分组。

### 4. 普通用户

只能访问首页和自己有权限的应用，不能访问后台接口。

## 二十一、待确认问题及建议结论

| 序号 | 待确认问题 | 建议 |
| --- | --- | --- |
| 1 | 应用管理员是否可以配置其他应用管理员？ | 允许。应用管理员可新增、删除、启用、停用自己负责应用下的应用管理员 |
| 2 | 应用管理员是否可以配置角色人员？ | 不允许。角色人员配置仅由系统管理员和业务管理员操作 |
| 3 | 业务管理员的人员管理范围如何控制？ | 一期按应用控制，二期按角色控制 |
| 4 | 业务管理员是否可以批量导入人员？ | 一期先不做，支持单个/批量勾选。二期支持 Excel 导入 |
| 5 | 配置人员后权限立即生效还是定时同步？ | 本系统内立即生效。目标系统取决于实时查询还是定时同步 |
| 6 | 自定义角色停用后已授权人员权限是否立即失效？ | 立即失效。角色停用后不能继续添加人员 |
| 7 | 应用删除采用物理删除还是逻辑停用？ | 逻辑停用，保留历史配置和关联数据 |
| 8 | ClientSecret 是否允许应用管理员查看或重置？ | 不允许。系统管理员可重置，重置后只展示一次并记录日志 |
| 9 | 应用分组由谁维护？ | 系统管理员统一维护，应用管理员只查看 |
| 10 | 首页是否按应用分组展示？ | 按分组展示，保留搜索，无分组放入"其他应用" |
| 11 | 用户基础信息从哪里获取？ | 从统一身份平台/HR/组织架构系统获取，本系统不维护用户主数据 |
| 12 | 业务管理员能否管理多个应用？ | 允许，每个应用单独配置授权关系 |
| 13 | 一个用户能否同时是多种管理员？ | 允许多身份叠加 |
| 14 | 多角色叠加时权限优先级如何处理？ | 后台按最高权限处理：系统管理员 > 应用管理员 > 业务管理员 > 普通用户。首页取权限并集 |
| 15 | 目标业务系统是否需要接收用户自定义角色信息？ | 建议。提供角色查询接口，目标系统自行判断 |
| 16 | 系统管理员首页是否展示全部应用？ | 只展示实际可访问应用。后台可管理全部 |
