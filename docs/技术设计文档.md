## 五、认证与权限设计

### 5.1 核心原则

**认证时一次性加载，后续全部走 Redis 缓存，权限变更时主动失效。**

### 5.2 Redis Key 设计

| Key | Value | TTL | 用途 |
|-----|-------|-----|------|
| `portal:token:{token}` | userId | 1800s | Token 到用户 ID 的映射，拦截器第一级缓存 |
| `portal:identity:{userId}` | 身份 JSON | 1800s | 用户身份信息，拦截器第二级缓存 |
| `portal:visible:apps:{userId}` | 可见应用 JSON Array | 1800s | 门户首页展示用 |

### 5.3 用户身份 JSON 结构

```
portal:identity:{userId}
```

```json
{
  "userId": "U001",
  "userName": "张三",
  "orgId": "NJ001",
  "orgName": "南京分行",
  "deptId": "D001",
  "deptName": "信息技术部",
  "isAdmin": true,
  "isSystemAdmin": true,
  "appAdminApps": [
    { "appCode": "salary-pay", "appName": "工资代发系统" }
  ],
  "bizAdminApps": []
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `isAdmin` | boolean | 只要 isSystemAdmin / appAdminApps 非空 / bizAdminApps 非空，任意为 true 则为 true。控制前端是否显示"后台管理"按钮 |
| `isSystemAdmin` | boolean | 查 system_admin 表 |
| `appAdminApps` | array | 查 app_admin 表，返回该用户作为应用管理员管理的应用列表 |
| `bizAdminApps` | array | 查 biz_admin 表，返回该用户作为业务管理员管理的应用列表 |

### 5.4 权限等级定义

| 等级 | 校验方法 | 可访问角色 | 说明 |
|---|---|---|---|
| A | `requireSystemAdmin()` | 仅系统管理员 | 新增应用、分组管理、系统管理员管理、重置密钥 |
| B | `requireAppAdmin(appCode)` | 系统管理员 + 应用管理员（限负责应用） | 应用配置、应用管理员/业务管理员/自定义角色配置 |
| C | `requireBizAdmin(appCode)` | 系统管理员 + 业务管理员（限管理应用） | 角色人员配置 |
| D | 数据范围过滤 | 三种管理员（数据范围不同） | 应用列表查询 |

### 5.5 权限校验矩阵

```

├──────────────────────────┼───────────┼──────────┼───────────┤
```

## 六、认证机制详解

### 6.1 三个服务的认证职责

| 服务 | 认证方式 | 保护路径 | 拦截器注册方式 |
|------|---------|---------|--------------|
| portal-service | 上游授权码登录 + `PORTAL_SESSION` + AuthInterceptor | `/api/auth/**`、`/api/portal/**` | Controller + Filter + `@Bean` MappedInterceptor |
| console-service | MappedInterceptor + AuthInterceptor | `/api/admin/**` | `@Bean` MappedInterceptor |
| server-service | FilterRegistrationBean + InternalTokenFilter | `/internal/*` | `@Bean` FilterRegistrationBean |

### 6.2 为什么 portal/console 用 Interceptor，server 用 Filter

- **portal / console**：需要验证用户 Token 并填充 UserContext，属于业务层认证，用 Spring MVC Interceptor 合适
- **server**：只需要验证调用方身份（是不是内部服务），不需要填充 UserContext，属于安全层防护，用 Servlet Filter 在最外层拦截

### 6.3 为什么用 MappedInterceptor 而不是 WebMvcConfigurer

WebMvcConfigurer 在 Spring MVC 初始化早期被收集，此时 Feign 客户端（依赖 MVC 的 HttpMessageConverters）还未就绪，会形成循环依赖。MappedInterceptor 是普通 Bean，在 MVC 初始化完成后通过 Bean 后处理注册，避开了这个问题。

### 6.4 为什么 AuthInterceptor 和 WebMvcConfig 分开

- `AuthInterceptor`（portal-common）：通用认证逻辑（从 `PORTAL_SESSION` 恢复 accessToken/idToken、查缓存、回源、填充上下文），不关心"怎么解析用户身份"
- `WebMvcConfig`（各服务）：定义"怎么回源初始化认证"（Lambda）和"拦截哪个路径"，是具体策略
- 这样 portal-service 和 console-service 可以复用同一个 `AuthInterceptor`，只在自己的 WebMvcConfig 中提供不同的路径配置

## 七、认证授权完整流程

### 7.1 请求处理全链路

上游点击“南京分行应用管理系统”后的默认入口：

```text
默认前端页面：/
默认初始化接口：GET /api/portal/init
登录成功默认回跳：/
```

`redirect_uri` 是注册给上游认证服务的回调地址，例如 `/api/auth/callback`；它和登录成功后回到的前端页面 `/` 不是同一个概念。

```
请求进入（携带 PORTAL_SESSION Cookie）
  │
  ├─ 1. 从 PORTAL_SESSION Cookie 读取 sessionId
  │     提取不到或 Redis session 不存在 → 抛 UnauthorizedException(401)
  │     命中 session → 恢复 accessToken + idToken
  │
  ├─ 2. 查 Redis: portal:token:{token} → userId
  │     ├─ 命中 → 继续
  │     └─ 未命中 → 携带 X-User-Token + X-ID-Token 调 Server /internal/auth/init 重新加载
  │
  ├─ 3. 查 Redis: portal:identity:{userId}
  │     ├─ 命中 → 放入 ThreadLocal 上下文，续期 TTL
  │     └─ 未命中 → 携带 X-User-Token + X-ID-Token 调 Server /internal/auth/init 重新加载
  │
  ├─ 4. 续期 portal:token + portal:identity + portal:visible:apps TTL（1800 秒）
  │
  ├─ 5. PermissionChecker 权限校验（仅 console-service）
  │
  ├─ 6. Controller 业务处理
  │
  └─ 7. afterCompletion → UserContext.clear()（清除 ThreadLocal）
```

### 7.2 缓存命中时的快速路径（大多数请求走这条路）

```
Token → Redis 查 userId（1 次 Redis 读）
      → Redis 查身份 JSON（1 次 Redis 读）
      → 反序列化填充 UserContext
      → 续期 TTL（3 次 Redis 写）
      → 放行
```

全程无远程调用，只查 2 次 Redis，写 3 次 Redis。

### 7.3 Server 认证初始化流程

```
POST /internal/auth/init
  → InternalTokenFilter：校验 X-Internal-Token 是否匹配配置密钥
  → AuthController：从 X-User-Token 读取用户 accessToken，从 X-ID-Token 读取用户 id_token
  → AuthServiceImpl：
      1. IdentityPlatformClient 解析 id_token → 获取 userId, userName, orgId, deptId
      2. 查 system_admin → isSystemAdmin
      3. 查 app_admin → appAdminApps
      4. 查 biz_admin → bizAdminApps
      5. 执行可见应用 SQL → visibleApps
      6. 计算 isAdmin = isSystemAdmin || appAdminApps非空 || bizAdminApps非空
      7. 写入 Redis 3 个 Key（TTL 1800 秒）
      8. 返回完整数据
```

### 7.4 PermissionChecker 权限校验

```
PermissionChecker.requireAppAdmin("APP01")
  → UserContext.get()
  → 如果 isSystemAdmin == true → 直接通过（系统管理员拥有所有权限）
  → 否则遍历 appAdminApps，看有没有 appCode == "APP01"
    → 有 → 通过
    → 没有 → 抛 ForbiddenException(403)
```

四种权限方法：

| 方法 | 含义 | 典型使用场景 |
|------|------|-------------|
| `requireSystemAdmin()` | 必须是系统管理员 | 新增应用、重置密钥 |
| `requireAppAdmin(appCode)` | 必须是该应用的应用管理员，系统管理员自动通过 | 编辑应用、启停应用 |
| `requireBizAdmin(appCode)` | 必须是该应用的业务管理员，系统管理员自动通过 | 管理角色人员 |
| `isAdmin()` | 是否任意管理员（不抛异常） | 控制前端"后台管理"按钮显示 |

权限等级关系：

```
系统管理员（isSystemAdmin）
  └─ 拥有所有权限，自动通过所有 requireXxx 校验

应用管理员（appAdminApps 包含某 appCode）
  └─ 拥有该应用的全部管理权限

业务管理员（bizAdminApps 包含某 appCode）
  └─ 拥有该应用的角色人员管理权限
```

### 7.5 InternalTokenFilter 内部接口保护

```
请求进入 server-service
  → InternalTokenFilter.doFilterInternal()
    → 路径以 /internal 开头？
      → 是：检查 Header "X-Internal-Token"
        → 值 == 配置的 portal.internal-token → 放行
        → 值不对或没有 → 抛 ForbiddenException(403)
      → 否：放行
```

### 7.6 异常处理

所有异常由 `GlobalExceptionHandler` 统一捕获，转换为 `Result<T>` 格式：

| 异常 | HTTP 状态码 | 触发场景 | 触发位置 |
|------|-----------|---------|---------|
| `UnauthorizedException` | 401 | 无 Token、Token 无效、Token 验证失败 | AuthInterceptor |
| `ForbiddenException` | 403 | 后台越权、内部 Token 校验失败、开放接口 Client 鉴权失败 | PermissionChecker / InternalTokenFilter / OpenApiService |
| `BusinessException` | 200 | 业务逻辑错误 | Service 层 |

## 八、缓存失效策略

### 8.1 失效场景

| 操作 | 需要失效的缓存 | 说明 |
|---|---|---|
| 添加/移除系统管理员 | `portal:identity:{被操作userId}` | 目标用户的身份变了 |
| 添加/移除应用管理员 | `portal:identity:{被操作userId}` | 目标用户的 appAdminApps 变了 |
| 添加/移除业务管理员 | `portal:identity:{被操作userId}` | 目标用户的 bizAdminApps 变了 |
| 给用户添加/移除自定义角色 | `portal:identity:{userId}` + `portal:visible:apps:{userId}` | 角色变了，可见应用可能变了 |
| 启用/停用应用 | `portal:visible:apps:*` | 影响范围大，清空所有用户的应用缓存 |
| 修改应用可见范围 | `portal:visible:apps:*` | 同上 |
| 启用/停用自定义角色 | `portal:visible:apps:*` | 角色停用影响所有拥有该角色的用户 |
| 应用分组变更 | `portal:visible:apps:*` | 分组变了，首页展示会变 |

### 8.2 失效方案

Console 做完写操作后，主动删 Redis Key：

```
Console 业务逻辑（写操作）
  │
  ├─ 执行数据库变更
  │
  ├─ 记录操作日志
  │
  └─ 失效缓存
      │
      ├─ 精确失效（影响单个用户）:
      │   DELETE portal:identity:{userId}
      │   DELETE portal:visible:apps:{userId}
      │
      └─ 批量失效（影响范围大）:
          SCAN portal:visible:apps:* → 批量删除
```

### 8.3 PermissionCacheManager 核心方法

| 方法 | 说明 | 使用方 |
|---|---|---|
| `getIdentity(userId)` | 读取用户身份缓存 | Portal、Console |
| `setVisibleApps(userId, apps)` | 写入可见应用缓存 | Server |
| `evictUser(userId)` | 精确失效：身份 + 可见应用 | Console |
| `evictAllVisibleApps()` | 批量失效：所有用户可见应用 | Console |
| `renewTTL(token, userId)` | 续期三个 Key 的 TTL | Portal、Console |

### 8.4 已知风险：`evictAllVisibleApps()` 全量清除

**现状：** 应用增删改、启停、分组变更时，`evictAllVisibleApps()` 通过 SCAN + UNLINK 清除所有用户的 `portal:visible:apps:*` 缓存。

**风险点：**

1. **爆炸半径过大** — 一次应用变更导致全部用户缓存同时失效。假设 10000 在线用户，就是 10000 条缓存同时被删。
2. **缓存击穿（Cache Stampede）** — 全量清除后，大量用户的下一次请求同时穿透到数据库重建缓存，产生瞬时查询峰值。
3. **SCAN 开销** — 用户量大时 SCAN 遍历本身消耗 Redis CPU，频繁触发时问题被放大。
4. **变更频率 × 用户量** — 每天变更 20 次、在线用户 5000，就是每天约 10 万次缓存重建。

**当前评估：** 用户量较小时可接受，缓存 TTL 仅 30 分钟，最差情况等于提前过期。

**后续优化方案（用户量上千或变更频率上升时）：**

- **版本号机制**：维护全局 `portal:apps:version` 计数器，应用变更时只 INCR，用户缓存写入时带版本号，读取时比对版本号，过期才重建。优点是 O(1) 失效、零 SCAN 开销、按需懒刷新、无击穿风险。
- **缩短 TTL + 放弃主动清除**：将 TTL 调至 5 分钟，不再主动清缓存。实现最简单，但存在最多 5 分钟数据延迟。

## 十、编码规范

### 10.1 注释

| 类别 | 写法 |
|------|------|
| 核心/复杂类 | 多行 Javadoc，说明职责和流程 |
| 简单 POJO/DTO | 单行 `/** ... */` |
| Controller 接口 | 每个接口 `/** 功能说明 */` |
| Service 方法 | 非显而易见逻辑才加 |
| 字段 | 含义不直观的加行尾注释 |
| 自解释的名称 | 不加注释 |

### 10.2 状态常量

所有状态字符串使用 `CommonConstant`，禁止硬编码：

```java
// ✅ 正确
.set(Entity::getStatus, CommonConstant.STATUS_ENABLED);

// ❌ 错误
.set(Entity::getStatus, "ENABLED");
```

已定义常量：
- `STATUS_ENABLED` / `STATUS_DISABLED`
- `SCOPE_APP` / `SCOPE_ROLE`
- `VISIBLE_ALL` / `VISIBLE_ROLE`
- `RESULT_SUCCESS` / `RESULT_FAIL`

### 10.3 审计字段

实体字段加 `@TableField(fill = FieldFill.INSERT/INSERT_UPDATE)`，Service 中不手动 set。由 `MyMetaObjectHandler` 自动填充：

| 字段 | 填充时机 |
|------|---------|
| `createBy` / `createName` / `createTime` | INSERT |
| `updateBy` / `updateName` / `updateTime` | INSERT + UPDATE |

### 10.4 缓存失效

每次写操作后调用 `PermissionCacheManager` 失效缓存：
- 影响单个用户 → `evictUser(userId)`
- 影响全部用户 → `evictAllVisibleApps()`

### 10.5 操作日志

增删改接口加 `@OperationLog` 注解：

```java
@OperationLog(module = "应用管理", type = "SAVE", desc = "新增/修改应用")
```

由 `OperationLogAspect` 自动拦截写入 `operation_log` 表。

### 10.6 统一响应

所有接口返回 `Result<T>`：

```java
return Result.success(data);       // 成功
return Result.error(500, "消息");   // 失败
return Result.success();           // 无数据成功
```

### 10.7 分页

```java
Page<Xxx> page = mapper.selectPage(new Page<>(pageNum, pageSize), wrapper);
return new PageResult<>(page.getTotal(), page.getRecords());
```

### 上游 Token 和 id_token 约定

当前链路分两段：

1. `portal-service` 对接上游授权码登录接口和 token 接口，使用 `code` 换取 `access_token` / `refresh_token` / `id_token`。
2. `server-service` 通过 `IdentityPlatformClient.verifyToken(accessToken, idToken)` 解析 `id_token` 获取用户基础信息，并加载本系统权限。

token 返回结构以 OIDC 字段为准：

```json
{
  "access_token": "xxx",
  "refresh_token": "xxx",
  "id_token": "xxx",
  "token_type": "Bearer",
  "expires_in": 10800
}
```

用户身份字段从 `id_token` 中提取，当前关注字段包括 `openId`、`userName`、`orgId`、`orgName`、`deptId`、`deptName`。生产环境需接入上游要求的 `SM2WithSM3` / `RS256` 验签实现后，再开启 `portal.upstream-auth.strict-id-token-verify=true`。

### 上游认证配置字段

| 配置项 | 所属服务 | 说明 |

| `portal.upstream-auth.authorize-url` | portal | 浏览器跳转的上游授权地址 |
| `portal.upstream-auth.token-url` | portal | 后端调用的上游 token 地址；UAT 文档要求使用 `http` |

| `portal.upstream-auth.private-key` / `public-key` | portal | 国密算法应用私钥/公钥，用于 token 接口签名 |
| `portal.upstream-auth.center-public-key` | server | 认证中心国密公钥，用于 `id_token` 验签 |
